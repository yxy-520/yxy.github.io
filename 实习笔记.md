  ```xml
  <select id="QueryPageFixedAssets" resultType="space.team.modules.project.vo.SmartProjectFixedAssetsVo">
      SELECT
          fa.serial_number,
          fa.name,
          fa.unit,
          fatc.name AS fixed_assets_type_code_name,
          d.name AS department_name
      	spfapd.smart_project_fixed_assets_plan_detailid AS smart_project_fixed_assets_plan_detailid
      FROM
          smart_project_fixed_assets_plan_detail spfapd
      LEFT JOIN fixed_assets fa
          ON spfapd.business_objectid = fa.fixed_assetsid
          AND fa.data_status_code = 61
      LEFT JOIN fixed_assets_type_code fatc
          ON fa.fixed_assets_type_code = fatc.fixed_assets_type_code
      LEFT JOIN department d
          ON fa.manage_departmentid = d.departmentid
          AND d.data_status_code = 61
      WHERE
          spfapd.data_status_code = 61
          AND spfapd.business_object_type_code = #{params.businessObjectTypeCode}
          <choose>
              <when test="params.smartProjectFixedAssetsPlanid != null">
                  AND spfapd.smart_project_fixed_assets_planid = #{params.smartProjectFixedAssetsPlanid}
              </when>
              <otherwise>
                  AND DATE_FORMAT(spfapd.month, '%Y-%m') = #{params.month}
                  AND spfapd.projectid = #{params.projectid}
              </otherwise>
          </choose>
  </select>
  ```

## 大文件上传

业务背景：前端上传文件包，后端负责将其落库`cms`内容库中，同时在华为云`obs`中备份，此外还需要将其落库`original`原始文件表中（会有一个文件状态字段记录文件：`0-待处理 1-处理中 2-未识别 3-待核验 4-已核验`，这个表除了上传文件包要落库，之后大模型解压后会将压缩包中的一个一个的文件也落库到`original`原始文件表中

数据库设计：
```sql
#sys_oss表 即存华为云obs的内容信息
此外还有关键字段file_md5码

#tb_sys_cms表 即文件内容表
该表的obs_id字段和sys_oss表的主键oss_id关联

#tb_biz_file_original表 即原始文件表
该表就的关键字段document_status即文件状态，大模型那边会定时查询该表中所有待处理（需要解压）、所有未识别（需要识别）......
```



这里和前端配合采用**分片上传**的策略（当然前端做了判断了，如果是小文件，直接调用原先的上传文件接口即可~）

1. 首先是根据`fileMd5`码（前端生成，文件唯一标识）判断文件是否存在，如果已经存在则直接秒传（即直接返回上传成功给前端即可，因为该文件已经在库中了），如果不存在则返回已上传的分片索引列表（这里应该是两种情况：1、根本就没有上传过分片；2、断点续传，由于存在网络波动导致只有部分分片上传成功，此时文件不会在库中，前端那里会只上传没有上传过的分片）

   ```java
   //controller:
   /**
    * 将上面两个接口合并，即根据fileMd5码判断这个文件是否已经存在 + 断点续传
    * 如果文件已存在 → 直接秒传
    * 如果文件不存在 → 返回已上传的分片索引列表（断点续传）
    */
   @GetMapping("/checkFileAndChunks")
   public R<Map<String, Object>> checkFileAndChunks(
       @RequestParam("fileMd5") String fileMd5,
       @RequestParam(value = "totalChunks", required = false) Integer totalChunks) {
   
       return iSysOssService.checkFileAndChunks(fileMd5, totalChunks);
   }
   
   //serviceImpl:
   /**
    * 根据fileMd5码判断这个文件是否已经存在 + 断点续传
    * 如果文件已存在 → 直接秒传
    * 如果文件不存在 → 返回已上传的分片索引列表（断点续传）
    */
   @Override
   public R<Map<String, Object>> checkFileAndChunks(String fileMd5, Integer totalChunks) {
       Map<String, Object> result = new HashMap<>(2);
   
       // Step1: 先查数据库，看整个文件是否已存在
       SysOss oss = baseMapper.selectOne(
           new LambdaQueryWrapper<SysOss>()
               .eq(SysOss::getFileMd5, fileMd5)
               .last("LIMIT 1"));
   
       if (oss != null) {
           // 文件已存在 → 秒传
           result.put("status", 1);
           result.put("uploadedChunks", Collections.emptyList());
           return R.ok(result);
       }
   
       // Step2: 文件不存在 → 检查已上传的分片
       ossConfigService.init();
       OssClient storage = OssFactory.instance();
   
       List<Integer> uploadedChunks = new ArrayList<>();
   
       if (totalChunks != null && totalChunks > 0) {
           for (int i = 0; i < totalChunks; i++) {
               String tempKey = "chunk/" + fileMd5 + "/" + i;
               try {
                   storage.getClient().getObjectMetadata(storage.getBucketName(), tempKey);
                   uploadedChunks.add(i);
               } catch (AmazonS3Exception e) {
                   if (e.getStatusCode() == 404) {
                       // 分片不存在 → 忽略
                   } else {
                       throw e; // 其他异常继续抛出
                   }
               }
           }
       }
   
       result.put("status", 0);  // 文件不存在
       result.put("uploadedChunks", uploadedChunks);
   
       return R.ok(result);
   }
   ```

2. 接着就是上传分片接口

   将分片传到华为云的临时目录（路径自己随便写），但是这里有个细节，**路径一定要拼`fileMd5`**，因为可能同时存在多人上传不同的文件，不同的文件对应的`fileMd5`肯定不同，所以不会存在分片冲突的情况，确保同一个文件的分片只会在一个临时目录下。**此外如果前端那边发现某个/些分片上传失败，它那边会捕获到，重新放入上传任务的队尾，重新上传。**

   注：这里还有点问题，路径拼`fileMd5`码还不够，因为如果多人同时上传相同文件（此时数据库中还没有该`md5`码），此时相当于在同一个临时目录下先后上传很多重复的分片（后覆盖前），虽然分片内容一样，覆盖了也不影响，但是之后若并发调用合并接口，先合并完的会直接清理临时目录下的分片，后合并的在合并过程中分片被清理了，此时文件是不完整的！所以最终的`obs/s3`协议会根据`ETag`校验每个分片是否完整、且分片是否齐全，此时校验不通过，无法将分片合并到指定目录下！

   所以一般来说我们的**临时目录是拼接上`uuid`，确保唯一性**，不过这样可能造成一个文件在多个目录下都存在，造成存储空间的浪费，此外也无法判断该文件是否正在被其他人上传（这里是上传中，不是上传过，因为如果上传过，则数据库中`oss`表就会有该`md5`码，此时直接秒传）。还有另外一种思路就是用**`redis`校验**，`key`就是`fileMd5`码（一般需要设置一个`ttl`，这里大概估计一下一般的大文件的上传时间），上传完才会删除`key`，这样确保多人上传同一个文件时，只会有一个正在上传，其余人轮询等待或者显示“文件正在上传中”，这样从根源上就避免了同一个文件被重复上传多次。这里`checkFileAndChunks`方法是没法判断多人同时上传同一个文件的，因为他只能判断一个文件已经上传成功落库了（即数据库中有这个`fileMd5`码）之后直接秒传，并不能解决多人同时上传同一个文件冲突问题！

   ```java
   //controller:
   /**
    * 上传分片
    * @param fileMd5 文件唯一标识
    * @param chunkIndex 分片序号
    * @param chunk 分片文件
    */
   @PostMapping("/uploadChunk")
   public R<Long> uploadChunk(String fileMd5,
                                Integer chunkIndex,
                                @RequestPart("file") MultipartFile chunk) throws IOException {
       return iSysOssService.uploadChunk(fileMd5, chunkIndex, chunk);
   }
   
   //serviceImpl:
   /**
    * 上传分片
    */
   @Override
   public R<Long> uploadChunk(String fileMd5, Integer chunkIndex, MultipartFile chunk) throws IOException {
       try{
           // 初始化redis默认配置文件上传地址
           ossConfigService.init();
           OssClient storage = OssFactory.instance();
   
           //OSS 临时目录：chunk/{fileMd5}/{chunkIndex}，避免本地存满盘
           String tempKey = "chunk/" + fileMd5 + "/" + chunkIndex;
   
           System.out.println("Uploaded chunk: " + tempKey);
   
           ObjectMetadata metadata = new ObjectMetadata();
           metadata.setContentLength(chunk.getSize());
           metadata.setContentType(chunk.getContentType());
   
           PutObjectRequest putObjectRequest =
               new PutObjectRequest(storage.getBucketName(), tempKey, chunk.getInputStream(), metadata);
           storage.getClient().putObject(putObjectRequest);
   
           return R.ok(Long.valueOf(chunkIndex));
       }catch (Exception e){
           // 失败：返回失败的分片号提示（带分片号，便于前端重试）
           return R.partFail( Long.valueOf(chunkIndex));
       }
   }
   ```

3. 最后是合并接口

   这里细节很多：

   目前是前端调用上传分片接口，将一个文件的所有分片上传到了华为云的临时目录中。

   合并时需要**先生成一个目标文件的路径**（这里文件尾缀如`.zip`要和源文件一致，而前缀需要唯一，避免相同文件名导致覆盖，一般用`uuid`即可）；然后会初始化分片上传的任务（注意：这里是指**将临时目录分片拷贝到目标文件路径**！）会返回一个`uploadId`，即这次"拼装任务"唯一标识，后续所有分片必须挂在该`uploadId`下，确保**合并过程中分片不会串文件**；接着就开始将临时分片逐个拷贝到目标文件路径中，且每成功拷贝一个分片就会返回一个`ETag`（分片的校验码），在最终调用`completeMultipartUpload`方法前**`obs/s3`协议会根据`ETag`校验每个分片是否完整、且分片是否齐全**，通过校验后才真正在目标文件路径中生成最终的文件。当然合并完成后会将临时分片清理掉，最后依次插入相关数据到`oss`表、`cms`表、`original`表

   ```java
   //controller:
   /**
    * 合并接口（所有分片都上传完成才调用该接口，这里由前端判断是否上传完所有分片）
    * @param fileMd5 文件唯一标识
    * @param fileName 文件名称（这里不再后端解析，由前端传）
    * @param suffix 文件名后缀（这里不再后端解析，由前端传）
    * @param totalChunks 总分片数
    * @return
    */
   @PostMapping("/mergeChunks")
   public R<Map<String, Object>> mergeChunks(@RequestParam("fileMd5") String fileMd5,
                                  @RequestParam("fileName") String fileName,
                                  @RequestParam("suffix") String suffix,
                                  @RequestParam("totalChunks") Integer totalChunks,
                                  @RequestParam("projectId") String projectId,
                                  @RequestParam("enterpriseId") String enterpriseId,
                                  @RequestParam("fileSize") String fileSize) {
       return iSysOssService.mergeChunks(fileMd5, fileName, suffix, totalChunks, projectId, enterpriseId, fileSize);
   }
   
   //serviceImpl:
   
   /**
    * 合并接口（所有分片都上传完成才调用该接口，这里由前端判断是否上传完所有分片）
    */
   @Override
   @Transactional(rollbackFor = Exception.class)
   public R<Map<String, Object>> mergeChunks(String fileMd5, String fileName, String suffix, Integer totalChunks, String projectId, String enterpriseId, String fileSize) {
       // 初始化redis默认配置文件上传地址
       ossConfigService.init();
       OssClient storage = OssFactory.instance();
       String path = storage.getPath(null, suffix);
       UploadResult uploadResult =
           UploadResult.builder().url(storage.getUrl() + "/" + path).filename(path).build();
   
       // 初始化分片上传
       InitiateMultipartUploadRequest initRequest =
           new InitiateMultipartUploadRequest(storage.getBucketName(), path);
       initRequest.setCannedACL(storage.getAccessPolicy().getAcl());
       InitiateMultipartUploadResult initResponse = storage.getClient().initiateMultipartUpload(initRequest);
       String uploadId = initResponse.getUploadId();
   
       List<PartETag> partETags = new ArrayList<>();
   
       try {
           for (int i = 0; i < totalChunks; i++) {
               String chunkKey = "chunk/" + fileMd5 + "/" + i;
   
               System.out.println("Copying chunk: " + chunkKey + " -> " + path);
   
               // 拷贝分片到最终文件（而不是先下载再上传）
               CopyPartRequest copyRequest = new CopyPartRequest()
                   .withSourceBucketName(storage.getBucketName())
                   .withSourceKey(chunkKey)
                   .withDestinationBucketName(storage.getBucketName())
                   .withDestinationKey(path)
                   .withUploadId(uploadId)
                   .withFirstByte(0L)
                   .withLastByte(storage.getClient().getObjectMetadata(storage.getBucketName(), chunkKey).getContentLength() - 1)
                   .withPartNumber(i + 1);
   
               CopyPartResult copyResult = storage.getClient().copyPart(copyRequest);
               partETags.add(copyResult.getPartETag());
           }
   
           // 完成合并
           CompleteMultipartUploadRequest compRequest =
               new CompleteMultipartUploadRequest(storage.getBucketName(), path, uploadId, partETags);
           storage.getClient().completeMultipartUpload(compRequest);
   
           // 清理临时分片
           for (int i = 0; i < totalChunks; i++) {
               String chunkKey = "chunk/" + fileMd5 + "/" + i;
               storage.getClient().deleteObject(storage.getBucketName(), chunkKey);
           }
   
           // 保存数据到oss中
           String url = uploadResult.getUrl();
           SysOss oss = new SysOss();
           oss.setUrl(url);
           oss.setFileSuffix(suffix);
           oss.setFileName(path);
           oss.setOriginalName(fileName);
           oss.setFileMd5(fileMd5);
           oss.setService(storage.getConfigKey());
   
           baseMapper.insert(oss);
   
           SysOssVo sysOssVo = new SysOssVo();
           BeanCopyUtils.copy(oss, sysOssVo);
   
           // 保存数据到cms中
           SysCms cmsEntity = new SysCms();
           try {
   
               SimpleDateFormat dateFormat = new SimpleDateFormat("/yyyy/MM/dd/");
   
               File folder = new File(serverConfig.getUploadFileSavePath().concat(dateFormat.format(new Date())));
               if (!folder.exists()) {
                   folder.mkdirs();
               }
   
               cmsEntity = cmsService.saveByOssFile(null, fileSize, fileName, sysOssVo, null, folder.getAbsolutePath(), suffix, String.valueOf(oss.getOssId()));
           } catch (Exception e) {
               e.printStackTrace();
               return R.fail("上传文件失败");
           }
   
           // 保存数据到file_original中
           fileOriginalInterFace.saveByCms(String.valueOf(cmsEntity.getCmsId()), fileName, enterpriseId, projectId);
   
           Map<String, Object> map = new HashMap<>(8);
           map.put("url", oss.getUrl());
           map.put("fileName", oss.getOriginalName());
           map.put("ossId", oss.getOssId().toString());
           map.put("cmsEntity", cmsEntity);
   
           return R.ok(map);
   
       } catch (Exception e) {
           storage.getClient().abortMultipartUpload(new AbortMultipartUploadRequest(storage.getBucketName(), path, uploadId));
           throw new ServiceException("合并失败: " + e.getMessage(), 500);
       }
   }
   ```

4. 未来优化点：

   现在是前端调用我后端的接口上传临时分片，这里涉及前端对后端的网络请求、后端对华为云的网络请求，如果一个`1G`左右的文件，通常分片大小都是`5MB`（兼容性最好），此时会有`200`个分片，即前端对后端的网络请求就高达`200`次，这肯定会拖慢整体的速度，如果前端能直传华为云（和后端沟通好，临时目录要一致，此外上传前先问后端要一个 **临时上传凭证（`STS Token` / 签名 URL）**，这里后端一次性生成所有分片的 **预签名 URL** 列表，前端拿到后直接把每个分片上传到华为云（`OBS`），从而减少后端中转和带宽压力。避免前端直连 `OBS` 时暴露主密钥），那么此时后端只需要提供根据`fileMd5`码判断这个文件是否已经存在 + 断点续传接口、合并接口即可。

```java
public R<Map<String, Object>> exportEquipmentWord(Long projectId) throws Exception {

    String projectName = Optional.ofNullable(projectMapper.getProjectName(projectId)).orElse("未知项目");

    String fileDir = serverConfig.getExportFileSavePath() + dateDir;
    Files.createDirectories(Paths.get(fileDir));

    // 2. 获取设备类资产数据
    List<AssetPart> items = equipmentMapper.getEquipmentsByProjectId(String.valueOf(projectId));
    List<String> htmlList = buildHtmlForAssets(items, EQUIPMENT);

    // 3. 分批渲染为多个临时 DOCX
    List<Path> tempDocs = new ArrayList<>();
    int index = 1;
    for (String html : htmlList) {
        Map<String, Object> data = new HashMap<>();
        data.put("content", html);

        Path subFile = Paths.get(fileDir, sanitizeFileName(projectName) + "_part" + index++ + ".docx");
        try (OutputStream fos = new FileOutputStream(subFile.toFile())) {
            poiTlHtmlExporter.renderHtmlToDocx(data, fos);
        }
        tempDocs.add(subFile);
        System.gc(); // 提示GC（可选）
    }

    // 4. 合并所有子文档为一个总文件
    String finalFileName = sanitizeFileName(projectName) + "-设备-资产明细表-" +
        LocalDate.now().format(DateTimeFormatter.BASIC_ISO_DATE) + ".docx";
    Path finalFile = Paths.get(fileDir, finalFileName);
    mergeDocxFiles(tempDocs, finalFile);

    SysCms cms = sysCmsService.saveExcelOrWordByFile(null, finalFile.toFile(), finalFileName, null, fileDir);
    return R.ok(CmsUtil.cmsToMap(cms));
}
```

```java
/**
 * 根据 items 组装 HTML 内容（包含各级标题和文件内容）
 */
private List<String> buildHtmlForAssets(List<AssetPart> items, String type) {
    List<String> htmlList = new ArrayList<>();

    if (items == null || items.isEmpty()) {
        return Collections.singletonList("<p>暂无数据</p>");
    }

    // 根据 contractId 分组
    Map<String, List<AssetPart>> groupedByContract = items.stream()
        .collect(Collectors.groupingBy(AssetPart::getContractId));

    // 遍历每个合同
    for (Map.Entry<String, List<AssetPart>> entry : groupedByContract.entrySet()) {
        StringBuilder html = new StringBuilder();

        String contractId = entry.getKey();
        List<AssetPart> parts = entry.getValue();

        // --- 设备或工程名称（多个时取第一个名称 + 等） ---
        String name;
        if (type.equals(EQUIPMENT)) {
            name = parts.get(0).getInvoiceEquipmentName();
        } else {
            name = parts.get(0).getInvoiceEngineeringName();
        }
        if (name == null) name = "未命名";
        if (parts.size() > 1) name = name + "等";
        html.append("<h5>").append(escapeHtml(name)).append("</h5>\n");

        // --- 合同 ---
        html.append("<h6>合同</h6>\n");

        List<String> contractFiles = contractMapper.getCmsFilePathsByContractId(Long.valueOf(contractId));
        if (contractFiles != null) {
            for (String path : contractFiles) {
                html.append(fileToHtmlUtil.createHtmlForFile(path));
            }
        }

        // --- 发票 ---
        // 先发票凭证，再发票文件
        html.append("<h6>发票</h6>\n");
        List<Map<String, Object>> invoiceWithVoucherFiles =
            invoiceMapper.getInvoiceWithVoucherFilesByContractId(contractId);
        if (invoiceWithVoucherFiles != null) {
            for (Map<String, Object> row : invoiceWithVoucherFiles) {
                String invoiceVoucherFile = (String) row.get("invoiceVoucherFile");
                String invoiceFile = (String) row.get("invoiceFile");
                if (invoiceVoucherFile != null) html.append(fileToHtmlUtil.createHtmlForFile(invoiceVoucherFile));
                if (invoiceFile != null) html.append(fileToHtmlUtil.createHtmlForFile(invoiceFile));
            }
        }

        // --- 付款 ---
        // 先付款凭证，再回单
        html.append("<h6>付款</h6>\n");
        List<Map<String, Object>> receiptWithVoucherFiles =
            receiptMapper.getReceiptsWithVoucherFilesByContractId(contractId);
        if (receiptWithVoucherFiles != null) {
            for (Map<String, Object> row : receiptWithVoucherFiles) {
                String receiptVoucherFile = (String) row.get("receiptVoucherFile");
                String receiptFile = (String) row.get("receiptFile");
                if (receiptVoucherFile != null) html.append(fileToHtmlUtil.createHtmlForFile(receiptVoucherFile));
                if (receiptFile != null) html.append(fileToHtmlUtil.createHtmlForFile(receiptFile));
            }
        }

        // --- 转固凭证 ---
        html.append("<h6>转固凭证</h6>\n");
        List<String> transferFiles = voucherDetailMapper.getCmsFilePathsByContractIds(Collections.singletonList(contractId));//这里我懒得改sql了。就单个查询就行
        if (transferFiles != null) {
            for (String path : transferFiles) {
                html.append(fileToHtmlUtil.createHtmlForFile(path));
            }
        }

        // --- 设备整体/铭牌图片 ---
        html.append("<h6>设备整体/铭牌图片</h6>\n");
        List<String> photoFiles = contractMapper.getPhotoFilePathsByContractId(Long.valueOf(contractId));
        if (photoFiles != null) {
            for (String path : photoFiles) {
                html.append(fileToHtmlUtil.createHtmlForFile(path));
            }
        }

        html.append("<hr/>"); // 每个合同后加分隔线
        htmlList.add(html.toString());
    }

    return htmlList;
}
```

```java
//renderHtmlToDocx方法
//POI-TL 导出包装
@Component
public class PoiTlHtmlExporter {
    @Autowired
    private ServerConfig serverConfig;
    public void renderHtmlToDocx(Map<String, Object> data, OutputStream out) throws Exception {
        // 关键：使用适配后的延迟加载策略
        HtmlRenderPolicy htmlPolicy = new LazyHtmlRenderPolicy();
        Configure config = Configure.builder().bind("content", htmlPolicy).build();

        try (
            // 根据环境初始化模板流（直接在try括号内处理）
            InputStream templateStream = serverConfig.getEnvironment().equals(EnumVars.ProjectEnvironment.DEV.getDesc())
                ? new org.springframework.core.io.ClassPathResource(EnumVars.ResourcesFileCode.WORD_EXPORT.getDesc()).getInputStream()
                : new FileInputStream(serverConfig.getResourcesTemplatePath() + EnumVars.ResourcesFileCode.WORD_EXPORT.getDesc());
            // 编译模板并渲染数据（作为第二个资源）
            XWPFTemplate template = XWPFTemplate.compile(templateStream, config).render(data)
        ) {
            // 3. 写入输出流
            template.write(out);
        }
    }
}
```

总之，采用分批导出的策略，以合同为基准，将对应`pdf`（先逐页渲染成图片）或图片进行`base64`编码并嵌入`html`（`<img>`标签）中，然后基于`poi-tl`将`html`渲染成小`word`（渲染完成后立即关闭流、触发 `GC`），最后使用 `Apache POI`操作 `XWPFDocument`，将各子文档内容合并成一个总`Word`，同时复制图片资源并重建关系（`rId` 映射），保证图片不会错乱。最后删除临时子文件，生成最终文档。
这里的复制图片资源并重建关系（`rId` 映射）：
因为多个子`word`的图片`id`可能冲突（重复），虽然它们引用的图片不同，这就导致最终合并时图片混乱。

## `mp`主键

数据库表对应的实体类的主键都会加上`@TableId`注解

![联想截图_20251022082345](D:\JAVA\shixi\实习笔记\photos\联想截图_20251022082345.png)

这里没有指定该注解的`type`属性值，**默认**就是`IdType.ASSIGN_ID`，即通过**雪花算法**分配主键值（这是`mybatis-plus`的方便之处！）。

雪花算法简单来说就是`[1位固定值0+41位时间戳][5位机器ID+5位服务id][12位序列号]`，高位的时间戳保证生成的主键`id`大体上是递增的，插入操作时接近“顺序插入”，减少索引页的分裂，提高性能（而像`uuid`无序的就不适合`mysql`的主键），中间的机器码（`5位机器ID+5位服务id`）能够支持部署`1024`个节点，而低位的序列号是支持`1ms`内生成`4096`个`id`。总之能够保证生成的主键`id`在分布式下的全局唯一性！

相较简单的主键自增策略，雪花算法最大的优势就是在分布式下（特别是水平分表）保证主键的唯一性，而主键自增是很容易在跨库时出现冲突的！

## `xxl-job`

`xxl-job`分布式任务调度，这里我用到了定时任务。

两个核心模块

- 调度中心

  就是`xxl-job`的任务管理后台，可以定义`cron`表达式、执行器、任务处理器...

- 执行器

  执行任务

在`java`中，配置文件中配置`xxl-job`地址、执行器名称等信息

```yaml
#这个知识样例，随便写的
xxl:
  job:
    admin:
      addresses: http://127.0.0.1:8080/xxl-job-admin
    executor:
      appname: demo-executor #执行器名称
      address: ''
      ip: ''
      port: 9999
      logpath: logs/xxl-job
      logretentiondays: 30
    accessToken: ''
```

然后编写任务代码

```java
@Component
public class SampleJob {
    @XxlJob("demoJobHandler")
    public void execute() throws Exception {
        System.out.println("XXL-JOB task running... " + new Date());
    }
}
```

注：这里的`demoJobHandler`是在执行器中注册的一个任务处理器

接着就在`xxl-job`任务管理后台中配置：

- 执行器，这里就是`demo-executor`，即我们的`springboot`项目，如果部署多个节点，则采取合适的路由策略路由，比如轮询，这样只会有一个执行器节点执行，或者特殊情况下广播，这样所有执行器节点执行，但这会造成重复执行！

- `JobHandler`即任务处理器，这里就是`demoJobHandler`

- `Cron`表达式，`0 0 9 * * ?`

  | 字段位置 | 含义                | 值   | 说明                       |
  | -------- | ------------------- | ---- | -------------------------- |
  | 第1位    | 秒（Seconds）       | 0    | 整秒开始                   |
  | 第2位    | 分（Minutes）       | 0    | 整分开始                   |
  | 第3位    | 时（Hours）         | 9    | 9 点执行                   |
  | 第4位    | 日（Day of month）  | *    | 每天                       |
  | 第5位    | 月（Month）         | *    | 每月                       |
  | 第6位    | 星期（Day of week） | ?    | 不指定（因为已指定“每天”） |

具体的业务就是：通过`xxl-job`做定时任务通知，每天晚上`18：29`检查所有员工昨天是否填写日报，若没有则调用企业微信`api`发送提醒
